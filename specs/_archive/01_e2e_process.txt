PROCESO 0 — CONTRATO DE ACTIVACION (cierra el agujero "config guardada pero NO publicada")

// [PROBLEMA] La causa-raiz mas comun del caso heroe es "config salva mas nao publicada": el restaurante
//   cree que su 2x1 esta listo porque lo configuro, pero NUNCA llego a la superficie que ve el consumidor.
//   Sin un objeto verificable que ate lo prometido a lo que el sistema debe chequear, no hay como prevenir por producto.
// [MEJORA] Toda promo persiste un "contrato de activacion" verificable que el sistema chequea ANTES del go-live (PE-ANTES-01).

[INICIO] // Un restaurante (o cuenta managed) crea o edita una promocion (ej.: 2x1, descuento).
  //   [C/analogia] Musixmatch: un sello agenda un release con credito de compositor en un DSP.

  [PASO 0.1] [ACCIoN] Persistir el [CONTENIDO] contrato_de_activacion como objeto versionado y verificable.
    -> [VARIABLE] estado_esperado // lo que el consumidor DEBERIA ver (publicada/activa, descuento %, mecanica 2x1, items elegibles)
    -> [VARIABLE] ventana_activacion // inicio/fin de la ventana en que la promo debe estar viva
    -> [VARIABLE] timezone_restaurante // huso del restaurante (no del sistema), para resolver la ventana correcta
    -> [VARIABLE] promo_id, id_restaurante
    -> [VARIABLE] feature_id // vinculo al evento de demanda (feature/email) si existe
    -> [VARIABLE] segmento_audiencia // [NUEVO] que cohort/region recibe el email; live fuera de ese segmento = dark-efectivo
    // [OBLIGACION] El contrato carga proveniencia por campo: input medido=[V], input estimado/modelado=[C].

  [PASO 0.2] [DECISIoN] ¿El contrato de activacion esta COMPLETO y sin ambiguedad?
    -> [SI] -> el objeto queda disponible para el pre-flight (PROCESO 1) y para el cierre por permanencia (PROCESO 3)
    -> [NO] -> [ACCIoN] marcar contrato_incompleto, NO inferir el campo faltante con suposicion
      -> [ROL] operador_1a10 completa/valida el campo faltante // [OBLIGACION] prohibida la suposicion no declarada (FD-07)

  [PASO 0.3] [SISTEMA] Cerebro_del_Cliente registra el episodio (estado+memoria) con timestamp y proveniencia.
    // [OBLIGACION] El contrato es FUENTE-DE-VERDAD del estado_esperado; jamas se compara contra cache de config.

PROCESO 1 — ANTES (pre-flight proactivo: 5 predicados config-vs-EN-VIVO, aviso, re-check, benchmark de cohort,

// [MEJORA] Mover la operacion de "remediar despues" a "prevenir en la raiz". La deteccion compara SIEMPRE
//   el estado CONFIGURADO contra el estado EN-VIVO en la superficie real del consumidor (no "config salva").
// [PROBLEMA] Un unico snapshot verde NO basta: hay regresiones despues del verde (despublicacion, aprobacion
//   expirada, huso recalculado). La vigilancia continua hasta el FIN del feature.

[INICIO] // Existe un evento de demanda agendado (feature/email Uber Eats) vinculado a una promo via calendario L-Edge,
  //   o una promo critica sin feature en ventana de riesgo. Arranca el agendamiento de checkpoints de pre-flight.

  [PASO 1.1] [SISTEMA] Programar cadencia de checkpoints proporcional a tiempo-hasta-pico y a $-en-juego (DET-02, SEG-R2).
    -> [VARIABLE] tier_efectivo // determina antelacion y densidad
    -> managed-brand -> primer checkpoint [C: T-72h] + re-verificacion continua hasta T-0
    -> managed-midmarket -> [C: T-48h] + T-24h + T-2h
    -> long-tail -> [C: T-24h] + T-2h
    // [OBLIGACION] El nivel de proteccion es proporcional al dinero arriesgado (LT-A1). Verde-SUSTENTADO, no verde-instantaneo.

  [PASO 1.2] [ACCIoN] En CADA checkpoint, leer el estado EN-VIVO directo de la fuente real de campanas/price-match (DET-01).
    [DECISIoN] ¿La lectura EN-VIVO es de calidad suficiente? (frescura <= TTL [C], fuente autoritativa respondio,
               payload no-ambiguo, tenant correcto) — gate de quality-of-information (DET-08), corre ANTES de todo juicio.
    -> [SI] -> avanzar a la comparacion de los 5 predicados (PASO 1.3)
    -> [NO] -> [ACCIoN] NO emitir 'verde' (falso-consuelo) NI 'dark' (falsa-alarma); emitir 3a categoria 'estado-no-verificable'
      -> [VARIABLE] estado_consumer = no_verificable
      -> [ROL] escalar a operador_1a10 con ultimo estado conocido [V] + timestamp + razon del fallo (FD-01)
      -> [SISTEMA] disparar retry idempotente con backoff + health-check de la integracion en paralelo
      // [OBLIGACION] Fail-open ('verde por ausencia de senal') esta PROHIBIDO. Estado-desconocido = estado-EN-RIESGO.

  [PASO 1.3] [DECISIoN] ¿El estado EN-VIVO DIVERGE del estado_esperado en al menos uno de los 5 predicados verificables? (DET-01)
    // (a) promo no publicada/inactiva en la superficie consumidor
    // (b) la ventana viva NO cubre la ventana del feature (inicio despues de T-0 o fin antes del fin del feature)
    // (c) terminos vivos (descuento %, mecanica 2x1, items elegibles) != terminos configurados
    // (d) timezone vivo resuelve la ventana equivocada vs. timezone_restaurante
    // (e) promo viva pero INVISIBLE al segmento_audiencia del email (live en region/cohort que no es el publico-objetivo)
    -> [NO] -> [ACCIoN] NO-AGIR / observar y re-vigilar (NBA-SEL-05); registrar snapshot verde con motivo
      // [OBLIGACION] NO-AGIR sobre estado-ya-verde tiene credito de valor CERO (no causo resultado; NBA-SEL-12).
    -> [SI] -> continuar; marcar cada divergencia con el eje (a..e) para ruteo de causa-raiz

  [PASO 1.4] [DECISIoN] ¿Es ruido que debe SUPRIMIRSE? (DET-03)
    // (a) feature cancelado/reagendado fuera de ventana, o restaurante removido del destaque
    // (b) promo off por INTENCION declarada y verificable (pausada, expiro legitimamente, borrador)
    // (c) restaurante con opt-out de contactos proactivos
    -> [SI, cancelado/intencional/opt-out] -> [ACCIoN] SUPRIMIR el trigger proactivo
      // [OBLIGACION] Supresion = 'no-habia-problema', NUNCA 'resuelto'. Jamas entra al numerador como valor.
      // En caso (c) se suprime SOLO la salida proactiva; el canal reactivo sigue intacto (HL-OPTOUT-01).
    -> [NO, o en duda] -> [ACCIoN] NO suprimir por defecto; emitir senal en severidad rebajada y rutear a humano
      // [PROBLEMA] 'Suprimir-en-la-duda' es el fallo MAS CARO (deja pasar promo-dark real = perdida de dinero).

  [PASO 1.5] [SISTEMA] Estabilizar antes de promover a SENAL (anti-flapping/debounce, DET-04).
    -> exigir [C: N=2] lecturas consistentes de 'dark' (o persistencia >= [C: M=10min]) antes de emitir
    -> EXCEPCION: si tiempo-hasta-pico < [C: 2h], emitir en la 1a lectura y verificar en la atencion
    // [OBLIGACION] Histeresis asimetrica: facil ABRIR cerca del pico, dificil CERRAR sin resultado atribuible (cierre = PROCESO 3 / DET-10).

  [PASO 1.6] [ACCIoN] Emitir el trigger 'promo-no-no-aire-pre-pico' y deduplicar (DET-06).
    -> colapsar todas las senales que comparten clave {id_restaurante + promo_id + feature/ventana} en UN caso, bajo lock idempotente
    -> [DECISIoN] ¿confianza_clasificacion >= piso_de_confianza[C] por tier? (DET-05)
      -> [SI] -> el caso entra a la cola con causa-raiz clasificada
      -> [NO] -> emitir 'sospecha-de-promo-dark-baja-confianza' ruteada OBLIGATORIAMENTE a humano, con [V] vs [I] separados

  [PASO 1.7] [SISTEMA] Calcular severidad y priorizar la cola de riesgo (DET-07, OP-A1).
    -> [VARIABLE] severidad = f($-en-riesgo modelado x probabilidad-de-seguir-dark x tiempo-hasta-pico x tier)
    // [OBLIGACION] Mostrar TODOS los inputs en la linea (nunca numero opaco).
    // [OBLIGACION] Separacion DURA: $-en-riesgo (priorizacion, puede ser declarado y cap-eado por proxy de cohort)
    //   vs. $-REALIZADO-confirmado (unico que entra al North Star, viene SOLO del cierre closed-loop, PROCESO 3).

  [PASO 1.8] [SISTEMA] Aviso PROACTIVO al dueno (LT-A2).
    [DECISIoN] ¿El restaurante tiene opt-out proactivo?
    -> [SI] -> suprimir el aviso, mantener deteccion registrada para el canal reactivo
    -> [NO] -> [CONTENIDO] enviar aviso en lenguaje que el dueno entiende, con 1 accion primaria de 1 clic + tiempo habil de corregir
      // [OBLIGACION] El sistema sigue vigilando hasta que el feature termine (no para en el aviso).

  [PASO 1.9] [ACCIoN] Correccion PROACTIVA en lote (motor de NBA + clamp de autonomia).
    [PASO 1.9.1] [SISTEMA] NBA mapea causa-raiz canonica -> tier-base pedido (AUT-13, MECE, fronteras disjuntas):
      -> (1) NO-PUBLICADA (job no corrio; config 1:1 con lo prometido) -> pide BAJO
      -> (2) HUSO/OFFSET (publico pero con ventana desplazada) -> pide BAJO si ventana no consumida, MEDIO si parcialmente consumida
      -> (3) APROBACION TRABADA en la cola -> pide MEDIO/ALTO (la IA NO auto-aprueba)
      -> (4) BUG DE ACTIVACION / regresion pos-snapshot (toggle/credito incorrecto en el DSP) -> pide MEDIO + abre candidato a spec
      -> (5) causa que implica DINERO/TERMINOS -> remite al techo-duro (PROCESO 4 / AUT-06), sin duplicar el gate
      -> (6) DESCONOCIDA -> siempre ALTO/escala
    [PASO 1.9.2] [SISTEMA] Calcular nivel efectivo = min(nivel_pedido_NBA, nivel_liberado_evals, techo_de_tier) (AUT-01, SEG-R3).
      -> [VARIABLE] nivel_efectivo
      // [OBLIGACION] Entrada nula/ausente/ilegible se sustituye por el tier MAS conservador ANTES del min(). Eval ausente != eval verde.
    [PASO 1.9.3] [DECISIoN] ¿nivel_efectivo = BAJO y la accion es reversible + idempotente bajo lock por id_restaurante? (AUT-04)
      -> [SI] -> [ACCIoN] ejecutar la correccion autonoma (reactivar/agendar toggle, corregir huso) + grabar before/after + proveniencia
        // [OBLIGACION] La autonomia NO puede auto-atestar 'resuelto'; eso pasa por verificacion de permanencia (AUT-15, PROCESO 3).
      -> [NO] -> [ACCIoN] enrutar a LOTE para aprobacion del operador (MEDIO) o escala-con-motivo (ALTO)
    [PASO 1.9.4] [ROL] operador_1a10 triada y decide en LOTE las correcciones propuestas (OP-A2).
      -> diff por item + 'aprobar-todo-menos-los-marcados'
      // [MEJORA] La leverage 1:10 viene de la triada en LOTE, no del clic individual (lo que se rompe en escala es la aprobacion caso-a-caso).

  [PASO 1.10] [SISTEMA] Benchmark de cohort como PRIOR SILENCIOSO (DET-09, SEG-R5, AI-ANTES-03).
    -> [VARIABLE] config_health_score // compara la config contra promos exitosas del MISMO cohort/tier (estructura, ventana, huso, aprobacion)
    [DECISIoN] ¿El desvio cruza el piso de severidad de DET-07?
    -> [NO] -> queda como PRIOR SILENCIOSO que eleva prioridad en el proximo pre-flight (NO contacto proactivo en masa)
      // [PROBLEMA] Pre-senales en masa entrenan a operador/cliente a ignorar alarmas -> protege la contra-metrica.
    -> [SI] -> escala como riesgo-de-config revisable por humano
    // [OBLIGACION] El cohort de comparacion lo fija una regla versionada; elegirlo ad-hoc para 'verse bien' es gaming auditable.

// [MEJORA] El valor solo cuenta si es CONFIRMADO + PERMANENTE + ATRIBUIBLE. Quedar LIVE nunca cierra el caso.

  //   (la fuente vuelve a LIVE-verificado y/o el pico paso) — DET-10, GNS-01..04.

    -> [SI] -> continuar; verificar a [C: 14 dias] que NO hubo reapertura/contacto-repetido por la misma causa
    // [OBLIGACION] Clasificador de 'misma-causa' con default CONSERVADOR: en la duda cuenta como MISMA causa (protege la contra-metrica).

  [PASO 3.3] [DECISIoN] ATRIBUIBLE — ¿el sinal de resultado volvio al CRM/Cerebro con la clave del caso (feature_id+promo_id+id_restaurante)? (GNS-02, ESC-R11)
    -> [NO] -> caso 'no-atribuible': cuenta 0 en el numerador (NUNCA valor estimado), aunque la promo haya quedado viva
      -> [SISTEMA] publicar la TASA de no-atribuible como senal de salud (no esconderla)
    -> [SI] -> el caso es elegible a entrar al numerador

  [PASO 3.4] [SISTEMA] ATRIBUCION POR SEGMENTO (sellos distintos en la UI).
    [DECISIoN] ¿el caso es long-tail o managed?
    -> long-tail -> atribucion por HOLDOUT (grupo de control), efecto medible estadisticamente

  [PASO 3.5] [SISTEMA] Calcular el North Star NETO (GNS-04).

    // [OBLIGACION] ANTI-HAND-WAVY: cualquier termino SIN modelo explicito (ej.: 'erosion de confianza') entra como 0-CON-FLAG,
    //   NUNCA un numero inventado. La sustraccion es idempotente por failure-event-id (anti double-count).
    // [OBLIGACION] El CALCULO de la sustraccion opera en MEDIO; cualquier EJECUCION que mueve $$ real es ALTO + HITL.

  [PASO 3.6] [SISTEMA] Cluster -> SPEC (loop soporte->producto, SS-R0..R8).
    [PASO 3.6.1] [DECISIoN] ¿el caso esta CERRADO con permanencia verificada y fuera de la ventana de resolucion activa? (SS-R0)

    [PASO 3.6.2] [DECISIoN] ¿el bucket cruza el threshold de evidencia? (SS-R2; gatillos OR + gate AND)
      // OR: frecuencia >= [C: N casos]; $-MEDIDO acumulado >= [C: R$ umbral]; un incidente catastrofico; bucket 'desconocida' >= [C: %]

      -> [NO] -> queda en 'observacion' con el item faltante explicito
      -> [SI] -> generar artefacto de spec (SS-R4): fix de RAIZ (no sintoma) + criterio-de-pronto testable + criterio de NO-regresion
    [PASO 3.6.3] [ROL] operador_1a10 / lider de producto APRUEBA la spec (SS-R6).
      // [OBLIGACION] La IA NUNCA auto-commitea cambio de producto/proceso; escala-con-motivo (ALTO). Toda spec aprobada recibe dueno nombrado [V].
    [PASO 3.6.4] [DECISIoN] ¿la spec shipo y el criterio-de-pronto se verifico en produccion sin empeorar la contra-metrica? (SS-R7)
      -> [SI] -> deflection BUENA: el cluster se marca RESUELTO; el valor reclamado es solo el INCREMENTAL (reduccion de frecuencia x $/caso evitado)
      -> [NO] -> deflection MALA: NO cierra el cluster ni cuenta valor; se PENALIZA restando el costo de la deflection-que-falla
    [PASO 3.6.5] [SISTEMA] Write-back del aprendizaje al Knowledge/cohort (SS-R8) tras pasar la suite de regresion (anti-envenenamiento).
      // El loop solo esta CERRADO cuando el aprendizaje esta EN EL AIRE para prevencion, no solo documentado.

  [PASO 3.7] [ACCIoN] 6o ACTO OFENSIVO — upsell -> Managed (capturar expansion, no solo evitar perdida).
    [DECISIoN] ¿del MISMO cohort hay cuentas P90+ listas para upsell?

    -> [NO] -> no forzar upsell; mantener el caso en su segmento

PROCESO 4 — GOBERNANZA / ANCHOR-CHECK EN RUNTIME (7 pasos), FAIL-CLOSED
===========================================================================================================

// [OBLIGACION] Este portal corre ANTES de CUALQUIER accion de la IA sobre una promo/release (proactiva o reactiva,
//   long-tail o managed). Es el portal inmutable que ningun componente puede contornar (HL-GATE-01, AI-GOV-01).
// [PROBLEMA] Sin politicas + context.md + knowledge la autonomia es teatro. El portal hace la autonomia VERDADERA y AUDITABLE.

[INICIO] // El motor de NBA produjo una accion candidata sobre una promo (reactivar toggle, corregir huso,
  //   escalar aprobacion, republicar campana, proponer compensacion) y va a despacharse.
  -> [VARIABLE] policy_version, autonomy_ceiling, decision_trace // se inicializa la traza de decision

  [PASO 4.0] [SISTEMA] CLASIFICAR (gate de elegibilidad + clasificacion de riesgo y causa).
    -> [DECISIoN] ¿pasan las 5 pre-condiciones del gate? (NBA-SEL-01)
      // (1) estado-de-verdad leido de la fuente EN-VIVO (no 'config salva')
      // (2) el feature AUN existe y no cambio de horario
      // (3) hay trinca de anclas
      // (4) la divergencia es REAL y >= piso de confianza [C: 0.7]
      // (5) NO hay edicion humana concurrente sobre el mismo id_restaurante/toggle (lock cooperativo)
    -> [NO] -> degrada a humano con motivo explicito ('live_state no leido' / 'evento cambio' / 'ancla X ausente' / 'edicion humana concurrente')
    -> [SI] -> [VARIABLE] risk_class asignada (reversibilidad, $-en-juego del EVENTO, tier/SLA leido de L3, irreversibilidad del dano)

  [PASO 4.1] [DECISIoN] POLITICA — ¿existe POLITICA aplicable versionada que autoriza la accion en ese tier? (parte 1 de la trinca, AUT-07, GNS-05)
    -> [NO] -> rebajar a ALTO/escala con motivo 'ancla politica ausente/divergente/stale'
    -> [SI] -> verificar tambien context.md + knowledge versionado (las 3 anclas se chequean A CADA accion, no una vez por sesion)
      // [OBLIGACION] Version-divergente/stale de cualquier ancla = ancla AUSENTE. Ancla-fantasma (entrada minima vacia) = ausente (FD-04).
      // [OBLIGACION] La correccion de la ancla, no solo su presencia, se atesta por eval del dominio (GNS-05).

  [PASO 4.2] [DECISIoN] HARD-LIMITS — ¿la accion viola algun techo-duro categorico? (evaluado ARRIBA y ANTES del min())
    -> [DECISIoN] ¿toca dinero/terminos comerciales? (clasificado por EFECTO, no por etiqueta de texto; HL-MONEY-01, AUT-06)
      -> [SI] -> FIJAR autonomy_ceiling = ALTO (escala-con-motivo); la IA PROPONE (valor, base de calculo, politica, before/after), NUNCA ejecuta $
        // [OBLIGACION] No existe camino en que evals 'liberen' dinero. Anti-fraccionamiento: N micro-pedidos correlacionados se SUMAN antes de comparar tetos.
      -> [NO] -> continuar
    -> [DECISIoN] ¿el acceso necesario cruzaria tenants o viola minimizacion de PII? (GOBERNANZA CROSS-TENANT, HL-TENANT-01, AUT-08, FD-09)
      -> [SI] -> BLOQUEAR la accion (fail-closed absoluto); dato de un restaurante NUNCA filtra a otro salvo como agregado de cohort anonimizado
        // [C/analogia] Musixmatch: Sony != Warner; cross-tenant es hard-no por GDPR/contrato. 'Necesite cruzar para resolver mas rapido' NO justifica violar aislamiento.
      -> [NO] -> continuar
    -> [DECISIoN] ¿la cuenta es managed-brand de alto valor (Beat C)? (HL-BEATC-01)
      -> [SI] -> techo estructuralmente rebajado: acciones mutantes >= MEDIO, todo toque comercial ALTO; diagnostico/propuesta siguen autonomos
      -> [NO] -> continuar
    -> [DECISIoN] ¿la accion es reversible + idempotente bajo lock por id_restaurante? (HL-IDEMP-01)
      -> [NO] -> no puede ser BAJO -> sube a MEDIO
      -> [SI] -> continuar

  [PASO 4.3] [DECISIoN] EVALS-GATE — ¿hay golden set valido y autonomia_liberada por capability/sub-causa? (EVPA-01..09)
    -> [DECISIoN] ¿existe golden set versionado con cobertura por causa-raiz [C: >=30 casos/causa; >=15% adversariales] + RED-TEAM SET?
      -> [NO] -> autonomia_liberada FIJADA en BAJO (o n/a para acciones de dinero); promocion bloqueada por construccion
      -> [SI] -> [VARIABLE] nivel_liberado = banda(score) POR SUB-CAUSA del caso (no el agregado; EVPA-06)
        // [OBLIGACION] El RED-TEAM SET garantiza que un juez co-sesgado no certifique el error (independencia juez<->humano mas alla de kappa).
        // [OBLIGACION] AUSENCIA de eval != eval verde -> se trata como liberado=humano (ALTO). Eval stale en runtime -> BAJO (fail-closed).
        // [OBLIGACION] Acciones de dinero/terminos NUNCA pasan de MEDIO por construccion, sin importar el score.

  [PASO 4.4] [DECISIoN] GROUNDING — ¿la accion esta fundamentada en estado EN-VIVO [V] y la confianza >= piso? (AUT-03, FD-01)
    -> [NO, estado no verificable o confianza < piso] -> autonomia maxima cae a ALTO/escala (confianza desconocida = confianza cero)
    -> [SI] -> continuar
    // [OBLIGACION] Fail-closed: jamas se actua sobre estado no verificado (actuar sobre un fantasma es la forma mas cara de errar).

  [PASO 4.5] [SISTEMA] MIN() — computar el nivel efectivo canonico (AUT-01, AI-GOV-01, EVPA-09).
    -> [VARIABLE] nivel_efectivo = MIN(nivel_pedido_NBA, nivel_liberado_evals, autonomy_ceiling)
    // [OBLIGACION] NUNCA el maximo, nunca la media. Cualquier entrada nula/ilegible se sustituye por el tier mas conservador ANTES del min().
    // [OBLIGACION] El par (nivel_pedido, nivel_liberado) se MUESTRA SIEMPRE al operador (nunca solo el resultado).
    // [OBLIGACION] Override humano solo puede REBAJAR; jamas elevar por encima de lo liberado por evals (AUT-11, EVPA-08).
    -> rutear: BAJO -> ejecuta+registra bajo lock | MEDIO -> lote p/aprobacion | ALTO -> escala-con-motivo (re-decision humana activa, anti-rubber-stamp ESC-R12)

  [PASO 4.6] [DECISIoN] TRACE — ¿la traza de auditoria COMPLETA puede grabarse? (sin log, sin accion; AUT-09, HL-ATTRIB-01)
    -> [NO] -> ABORTAR la accion y degradar a humano (accion sin traza = accion no-autorizada)
    -> [SI] -> [ACCIoN] grabar [VARIABLE] decision_trace:
      -> nivel_pedido, nivel_liberado, nivel_efectivo, risk_class, confianza, anclas (versiones), before/after, proveniencia, actor, motivo
      -> [VARIABLE] costo_ia_por_decision // registrar el costo de IA por decision (unit-economics; leccion €3 -> €1, margen = moat economico)
    // [OBLIGACION] BAJO es literalmente 'ejecuta+REGISTRA'; ALTO es 'escala CON MOTIVO'. Sin motivo registrado no es una escala valida.

  [PASO 4.X] [SISTEMA] DINaMICA fail-closed (monitor continuo, transversal a los 7 pasos):
    -> [DECISIoN] ¿evals detectan regresion por sub-causa, o la contra-metrica cruza umbral malo? (AUT-10, EVPA-05, HL-FREEZE-01)
      -> [SI] -> REBAJAR automaticamente nivel_liberado para la(s) sub-causa(s) afectada(s) (granular, no el dominio entero)
        -> en breach grave [C: contra-metrica > 2x baseline o falso-'resuelto' en caso de dinero] -> FREEZE de mutaciones (kill-switch)
        -> re-elevar exige eval-verde + firma humana + dwell + canary con atribucion madura (EVPA-04); el loop NUNCA auto-re-eleva
      -> [NO] -> mantener nivel vigente
    // [OBLIGACION] Si el propio guardian (eval/monitor/anchor-checker) esta off o ambiguo -> se asume el peor caso y se rebaja.
    //   La ausencia del guardian NUNCA libera el portal. PISO REACTIVO: nunca dejar el inbound del cluster sin atencion humana (EVPA-05).
